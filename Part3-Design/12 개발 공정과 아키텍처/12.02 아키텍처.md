# 아키텍처

아키텍처(Architecture)는 소프트웨어 시스템, 건물, 또는 조직과 같은 복잡한 구조체의 기본 구성 요소(Elements), 이들 간의 관계(Relationships), 그리고 이들을 설계하고 진화시키는 원칙과 가이드라인을 정의하는 총체적인 개념입니다.

쉽게 말해, 아키텍처는 시스템의 큰 그림을 그린 설계도이며, '무엇을(What)', '어떻게(How)', '왜(Why)' 그렇게 구성했는지에 대한 결정의 집합입니다.

## 소프트웨어 아키텍처의 핵심

소프트웨어 아키텍처는 시스템이 충족해야 하는 기능적 요구사항뿐만 아니라, 비기능적 요구사항(품질 속성)을 달성하는 데 중점을 둡니다.

### 구성 요소 (Elements)
시스템을 이루는 주요 모듈, 컴포넌트, 서비스, 데이터베이스, 사용자 인터페이스 등입니다. (예: `UserService`, `PaymentGateway`, `Database`)

## 관계 (Relationships)
구성 요소들이 서로 어떻게 상호작용하고 통신하는지 정의합니다. (예: `REST API 호출`, `메시지 큐 통신`, `데이터베이스 연결`)

### 원칙 (Principles)
아키텍처를 설계할 때 따르는 규칙과 제약 조건입니다. (예: `보안성`, `확장성`, `유지보수성`)

## 아키텍처의 중요성 (품질 속성)

훌륭한 아키텍처는 시스템의 품질 속성(Non-Functional Requirements)을 보장하는 기반이 됩니다.

* 확장성 (Scalability): 사용자 증가나 부하 증가에 대비하여 시스템을 쉽게 확장할 수 있는 능력.
* 성능 (Performance): 시스템이 작업을 효율적으로 처리하는 속도와 응답 시간.
* 유지보수성 (Maintainability): 시스템을 쉽게 수정, 개선, 디버깅할 수 있는 능력.
* 신뢰성 (Reliability): 시스템이 오류 없이 의도한 대로 일관되게 작동하는 능력.
* 보안성 (Security): 시스템과 데이터가 무단 접근으로부터 보호되는 정도.

## 대표적인 아키텍처 패턴

소프트웨어 시스템을 조직하는 데 널리 사용되는 몇 가지 주요 패턴입니다.

| 패턴                                    | 설명                                                                                                   | 적합한 상황                                                       |
| :-------------------------------------- | :----------------------------------------------------------------------------------------------------- | :---------------------------------------------------------------- |
| 단일형 아키텍처 (Monolithic)            | 모든 구성 요소가 단일 코드베이스와 단일 서비스로 묶여 배포되는 전통적인 방식.                          | 소규모 프로젝트, 단순한 애플리케이션, 빠른 초기 개발이 필요할 때. |
| 계층형 아키텍처 (Layered, N-Tier)       | 시스템을 역할별로 수평적인 계층(프레젠테이션, 비즈니스 로직, 데이터 접근)으로 나누는 방식. (가장 흔함) | 엔터프라이즈 애플리케이션, 내부 비즈니스 로직이 명확할 때.        |
| 마이크로서비스 아키텍처 (Microservices) | 시스템을 독립적으로 배포 가능한 작고 느슨하게 결합된 서비스들의 집합으로 구성하는 방식.                | 대규모, 고도화된 확장성이 필요한 서비스(넷플릭스, 아마존 등).     |
| 이벤트 중심 아키텍처 (Event-Driven)     | 컴포넌트들이 이벤트를 발행하고 구독함으로써 비동기적으로 통신하는 방식.                                | 실시간 처리, 복잡한 데이터 파이프라인, 낮은 지연시간이 필요할 때. |

## 5-Step 아키텍처

### 5-Step Architecture 소개

5-Step Architecture는 헥사고날 아키텍처(Hexagonal)와 도메인 주도 설계(DDD)를 기반으로 한 소프트웨어 제조 공정 표준입니다. 서비스는 로직을 가지지 않는 흐름의 조율자(Functor)이며, 모델은 비즈니스 규칙을 가진 두뇌(Brain) 역할을 수행합니다. 각 도메인은 철저히 격리되어 인터페이스 계약으로만 협업합니다. 이는 모듈형 아키텍처를 지향합니다.


### 1. 표준 개발 공정 (The 5-Step Standard Process)

* [Step 1] 도메인 자아 확립 (Domain Model Discovery)
    * 비즈니스의 핵심인 Aggregate Root(AR)를 정의하고 공통 `BaseEntity`를 상속받아 정체성을 부여합니다.
    * 생성자 공정을 신규 생성(Creation)과 데이터 복원(Reconstitution, `reconstitute`)으로 철저히 이원화합니다.
    * 상태 변경 시 이력 기록(`recordModification`)을 강제하여 정합성을 방어합니다.


* [Step 2] 법전 정의 (Policy Abstraction)
    * 보안, 계산 규칙 등 외부 환경에 따라 변하기 쉬운 비즈니스 제약을 `@FunctionalInterface` 등으로 추상화합니다.
    * 모델은 법전(Policy)의 상세 구현을 모른 채, 주입된 법에 의해서만 판단하도록 설계합니다.


* [Step 3] 흐름 조율 (Service Orchestration)
    * 서비스는 로직을 소유하지 않고 전체 공정의 흐름(Activity 선언 -> AR 복원 -> Policy 주입 및 실행 -> 저장)만 관장합니다.
    * 요구사항 추적을 위해 모든 Activity에 Usecase ID 명시를 강제하며, 단일 로컬 트랜잭션 경계를 확정합니다.


* [Step 4] 도구 제작 (Activity Implementation)
    * Step 3에서 도출된 인터페이스의 구체적인 인프라 기술 도구를 구현합니다.
    * 명세(Port)는 비즈니스 패키지(`activity`)에, 구현체(Actions, Adapter)는 인프라 패키지(`infra.*`)에 두어 컴파일 시점 의존성을 격리합니다.
    * 어댑터가 기술 데이터와 도메인 모델 간의 차이를 메우는 매핑 책임을 전담합니다.


* [Step 5] 반복 공정 검증 (Multilayered Testing)
    * Model Test: 정책을 Stub/Mock으로 주입하여 순수 비즈니스 로직을 ms 단위로 가볍고 빠르게 검증합니다.
    * Service Test: 컴포넌트 간의 호출 흐름(Orchestration)만 검증하여 테스트가 무거워지는 것을 방지합니다.


### 2. 주요 구현 가이드라인 (Implementation Guidelines)

* 통합 복원 (`reconstitute`): 생성과 조회를 도메인 관점에서 하나로 통합하여 기술 환경에 구애받지 않고 모델을 복원합니다.
* 명시적 정책 주입: AR의 메서드가 정책을 필요로 할 때 서비스가 파라미터로 명시적으로 전달(법전 주입)합니다.
* 방어적 복사 (Defensive Copy): 내부 컬렉션 노출 시 `Collections.unmodifiableList()` 등을 사용하여 외부 오염을 원천 차단합니다.
* 응답/예외 표준화: `DefaultResponseHandler`를 통해 모든 응답을 `Success` 또는 `Fail` 구조로 자동 캡슐화합니다.
* 슬라이스 테스트 & 문서화 자동화: `MockMvcTester`와 Spring Rest Docs를 결합하여 테스트를 통과한 케이스만 API 명세서(`.adoc`)로 자동 생성되도록 강제합니다.

## 5-Step Architecture가 효과적인 아키텍처적 맥락

5-Step Architecture는 도메인의 복잡성이 높고, 비즈니스 규칙이 빈번하게 변경되거나 확장되는 프로젝트에서 가장 강력한 효과를 발휘합니다. 구체적으로 다음과 같은 상황과 요구사항이 있을 때 이 구조를 도입하는 것이 적합합니다.

### 1. 도메인 규칙 및 생애주기가 매우 복잡한 상황

* 복잡한 상태 머신(State Machine)이 필요한 도메인: 주문, 배송, 금융 계약, 승인 결재 프로세스처럼 상태(Status)가 많고, 특정 이벤트(Event)와 제약 조건(Policy)에 의해서만 상태가 전이되어야 하는 시스템에 적합합니다.
* 비즈니스 규칙과 기술 인프라의 분리가 절실할 때: 데이터베이스(JPA/JdbcClient), 메시지 큐(Kafka), 캐시(Valkey) 같은 기술적 선택이 도메인 비즈니스 논리에 영향을 주지 않아야 하는 고도화된 엔터프라이즈 시스템에 적합합니다.

### 2. 비즈니스 정책(Policy)이 빈번하게 변경되는 상황

* 환경이나 조건에 따라 법전(규칙)이 바뀌는 도메인: 예를 들어, 회원 등급별 할인 정책, 국가별 세금 계산법, 기간별 프로모션 규칙 등 '핵심 흐름은 같지만 세부 계산/제어 로직이 계속 바뀌는 경우'에 적합합니다.
* Step 2의 법전 정의(Policy Abstraction)를 통해 규칙을 인터페이스로 분리해 두면, 도메인 모델(Brain)이나 서비스(Functor)의 수정 없이 새로운 정책 구현체만 주입하여 요구사항 변화에 유연하게 대응할 수 있습니다.

### 3. 마이크로서비스(MSA) 또는 모듈형 모놀리스(Modular Monolith) 구조

* 철저한 모듈 격리가 필요할 때: 가이드에서 명시한 것처럼 각 도메인 모듈이 '격리된 자아'를 가지고 인터페이스 계약으로만 소통해야 하는 구조에 최적화되어 있습니다. 특히 Spring Modulith 등을 활용하여 모듈 간 컴파일 시점의 의존성을 강력하게 제어하고 싶을 때 매우 유용합니다.

### 4. 요구사항 추적성(Traceability)과 데이터 정합성이 필수적인 시스템

* 감사 추적(Audit Trail)이 중요한 도메인: 금융, 이커머스 등 "누가, 언제, 왜 이 데이터를 변경했는가"를 명확히 추적해야 하는 시스템에 적합합니다. Step 1에서 `recordModification(operator)`을 통해 모델의 자아 변화 이력을 내재화하므로 정교한 이력 관리가 가능합니다.
* 유스케이스 중심의 추적: Step 3/4에서 모든 Activity에 Usecase ID 명시를 강제하므로, 기획 요구사항(Usecase)과 실제 구현 코드 간의 1:1 추적성이 명확히 확보되어야 하는 대규모 프로젝트에 유리합니다.

### 5. 테스트 비용을 낮추고 신뢰성을 높여야 하는 대규모 팀

* 비즈니스 로직을 검증할 때 인프라(DB 등)를 띄우지 않고 오직 순수 Java 모델과 Mock 정책만으로 ms 단위의 빠른 단위 테스트(Model Test)를 반복하고 싶을 때 적합합니다.
* 슬라이스 테스트 환경이 표준화되어 있어, 여러 개발자가 동시에 작업해도 테스트 신뢰성을 유지하며 지속적인 통합(CI)을 매끄럽게 수행할 수 있습니다.

---

> 反 패턴 (적합하지 않은 상황):
> 단순한 CRUD(등록, 수정, 삭제, 조회) 위주의 시스템이나 도메인 로직이 거의 없는 단순 관리자 페이지, 프로토타입 개발 등에서는 5-Step Architecture가 오히려 과도한 엔지니어링(Over-engineering)이 되어 생산성을 떨어뜨릴 수 있습니다.
