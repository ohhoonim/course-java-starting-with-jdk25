# 👤 중급 개발자 : 김설계 (30세, 4년 차 백엔드 엔지니어)

> "구현하지 못하는 기능은 없습니다. 하지만 '이 코드가 최선인가? 1년 뒤에도 유지보수할 수 있을까?'라는 질문 앞에서는 자꾸 작아집니다."

### 1. 프로필 개요

* 직무: 이커머스 스타트업 백엔드 개발자
* 경력: 만 4년 (주니어 티를 벗고 본격적으로 핵심 시스템을 담당하는 시기)
* 주요 기술 스택: Java 17, Spring Boot, Spring Data JPA, MySQL, Redis, AWS
* 현재 상태: 요구사항을 코드로 바꾸는 '기능 구현' 속도는 매우 빠름. 하지만 시스템이 커지면서 레거시 코드의 늪에 빠지기 시작했고, 이를 해결할 '설계 패러다임'의 필요성을 절실히 느끼는 중.

### 2. 행동 패턴 및 보유 역량

* 기능 중심적 사고: 서비스 초기에는 컨트롤러-서비스-레포지토리 형태의 CRUD 기반 빈약한 도메인 모델(Anemic Domain Model)로 빠르게 아웃풋을 냈음.
* 성장 통과의례: 비즈니스 루틴이 복잡해질 수록 서비스 클래스가 수천 줄로 비대해지며(Fat Service), 사소한 수정에도 사이드 이펙트가 발생해 QA 단계에서 버그가 터지는 경험을 자주 함.
* 독학의 한계: 오브젝트, DDD 조영호 님 저서 등을 읽으며 SOLID 원칙과 디자인 패턴을 머리로는 이해했으나, "내 프로젝트의 이 복잡한 도메인 상태 변화를 자바 코드로 어떻게 정밀하게 녹여내야 하는가?"에 대한 실무적 연결 고리를 찾지 못함.

### 3. 좌절 요인 (Pain Points)

* 잦은 요구사항 변경: 사용자와 기획자의 잦은 기능 수정 및 추가 요청이 올 때마다 설계가 엎어지고 누더기 코드가 됨.
* 비동기 소통의 부재: 명확한 명세서(Graph 단계) 없이 구두나 단편적인 텍스트로만 소통하다 보니 대면 회의가 길어지고 개발 생산성이 저하됨.
* 추상화에 대한 두려움: 자바 최신 스펙(`sealed`, `record`, `switch pattern matching`)이나 아키텍처 제약 조건을 활용해 도메인을 화이트박스 모델링하는 정교한 설계 경험이 부족함.

### 4. 학습 목표 및 니즈 (Goals)

* 리치 도메인 모델(Rich Domain Model) 구축: 비즈니스 로직과 상태 전이 제약 조건을 도메인 엔티티와 값 객체(VO) 내부로 격리하여 안전한 캡슐화를 달성하고 싶음.
* 시니어 수준의 시각 확보: 유스케이스, 활동, 상태 기계 다이어그램 등 UML을 단순 시각 자료가 아닌, '코드를 도출하기 위한 정밀한 명세서'로 다루는 아키텍처적 시야를 갖고 싶음.
* 지속 가능한 아키텍처 설계: 컴파일 타임에 비즈니스 룰의 완전성(Exhaustiveness)을 검증할 수 있는 견고한 코드 작성법을 마스터하여 기술 부채를 통제하고 싶음.

### 5. 선호하는 학습 스타일

* 추상적이고 원론적인 소리(예: "결합도를 낮추고 응집도를 높이세요")를 극도로 싫어함.
* 다이어그램(시각적 정체성)이 실제 자바 코드(구조적 명세)로 어떻게 1:1 매핑되는지 보여주는 '규칙 기반 매핑 가이드'나 '아키텍처 스킬 문서' 패턴에 열광함.
* 본인이 짠 코드를 리팩토링해가며 설계 원칙이 적용되었을 때의 전후 생산성 차이를 눈으로 확인하는 화이트박스 형태의 실습을 선호함.
