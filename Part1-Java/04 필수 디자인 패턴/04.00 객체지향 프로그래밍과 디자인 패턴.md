## 객체지향 프로그래밍 (Obejct Oriented Programming)

- [영상 : 객체지향 프로그래밍](https://youtu.be/clRVWo7NBZ0)

### OOP 주요 개념
- 캡슐화 (Encapsulation): 데이터와 기능을 하나로 묶고 외부 접근을 제한하여 정보를 보호함.
- 상속 (Inheritance): 기존 클래스의 특성을 이어받아 재사용하고 기능을 확장함.
- 추상화 (Abstraction): 복잡한 내부 구현은 숨기고 핵심적인 개념(공통점)만 추출하여 정의함.
- 다형성 (Polymorphism): 하나의 객체나 메서드가 상황에 따라 여러 가지 형태로 동작함.

### SOLID: OOP 설계원칙
- SRP (단일 책임 원칙): 클래스는 하나의 기능만 수행하며, 변경 이유는 단 하나여야 함.
- OCP (개방-폐쇄 원칙): 확장은 가능하되 기존 코드 수정은 최소화해야 함.
- LSP (리스코프 치환 원칙): 자식 클래스는 언제나 부모 클래스를 대체할 수 있어야 함.
- ISP (인터페이스 분리 원칙): 사용하지 않는 메서드에 의존하지 않도록 인터페이스를 작게 쪼개야 함.
- DIP (의존 역전 원칙): 구체적인 클래스가 아닌 인터페이스나 추상화에 의존해야 함.

### OOP 디자인 패턴 요약

* 생성 패턴 (Creational): 객체 생성 방식의 효율성과 유연성을 높임 (Singleton, Factory, Builder 등).
* 구조 패턴 (Structural): 클래스나 객체를 조합하여 더 큰 구조를 형성 (Adapter, Decorator, Proxy 등).
* 행위 패턴 (Behavioral): 객체 간의 책임 분배와 통신 방식을 정의 (Strategy, Observer, State 등).

### 1. 생성 패턴 (Creational Patterns)
객체의 생성 과정을 추상화하여 시스템이 객체의 생성, 구성, 표현 방식에 독립적일 수 있게 합니다.

* Singleton: 클래스의 인스턴스가 딱 하나만 생성되도록 보장합니다.
* Factory Method: 객체 생성 처리를 서브 클래스로 분리하여 캡슐화합니다.
* Abstract Factory: 연관된 객체들의 가족을 생성하기 위한 인터페이스를 제공합니다.
* Builder: 복잡한 객체의 생성 단계와 표현 단계를 분리하여 동일한 생성 공정에서 다양한 결과물을 만듭니다.

### 2. 구조 패턴 (Structural Patterns)
클래스나 객체를 조합하여 더 큰 구조를 만드는 패턴입니다. 서로 독립적으로 개발된 클래스들이 함께 작동할 수 있게 돕습니다.

* Adapter: 호환되지 않는 인터페이스를 가진 클래스들이 함께 작동할 수 있도록 변환해줍니다.
* Decorator: 객체에 동적으로 새로운 책임(기능)을 추가합니다. 상속보다 유연한 기능 확장이 가능합니다.
* Proxy: 다른 객체에 대한 접근을 제어하기 위해 대리자 객체를 둡니다. (지연 로딩, 보안 등)
* Facade: 복잡한 서브시스템에 대해 단순화된 통합 인터페이스를 제공합니다.

### 3. 행위 패턴 (Behavioral Patterns)
객체들이 상호작용하는 방법과 책임을 분산하는 방법을 정의합니다. 결합도를 낮추고 유연한 통신을 가능하게 합니다.

* Strategy: 알고리즘 군을 정의하고 각각을 캡슐화하여 실행 중에 선택할 수 있게 합니다.
* Observer: 한 객체의 상태 변화가 있을 때 의존하는 다른 객체들에게 자동으로 통지합니다.
* State: 객체의 내부 상태에 따라 스스로 행동을 변경할 수 있게 합니다.
* Template Method: 알고리즘의 골격을 상위 클래스에서 정의하고, 세부 단계는 하위 클래스에서 구현합니다.
