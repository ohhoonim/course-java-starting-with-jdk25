# 04.03 Strategy 패턴

## 실습 

### 실습 1) SOLID 설계 원칙

- 객체지향 프로그래밍 SOLID설계원칙에 대해 조사하시오. 
- 각각의 설계원칙에 대한 위반 사례와 준수 사례에 대해 코드를 작성하시오.(AI 챗 사용)

> 📋 SOLID 설계 원칙 
> 
>- S: SRP (Single Responsibility Principle) - 단일 책임 원칙
>    - 핵심: 한 클래스는 오직 하나의 책임만 가져야 하며, 그 책임과 관련된 하나의 이유로만 변경되어야 한다.
>    - 목표: 클래스의 변경 사항이 파급 효과를 일으키는 것을 방지하고, 클래스 응집도를 높인다.
>    - 예시: 사용자 정보 관리 클래스는 데이터베이스 저장(Persistence)과 사용자 정보 출력(Presentation)을 동시에 처리하지 않고, 각각 별도의 클래스로 분리한다.
>
>- O: OCP (Open/Closed Principle) - 개방/폐쇄 원칙
>    - 핵심: 소프트웨어 요소(클래스, 모듈 등)는 확장에는 열려 있어야 하고, 변경에는 닫혀 있어야 한다.
>    - 목표: 기존 코드를 수정하지 않고도 새로운 기능(행동)을 추가할 수 있도록 시스템을 유연하게 만든다.
>    - 구현 방법: 주로 인터페이스나 추상 클래스를 사용하여 다형성(Polymorphism)을 확보한다 (Strategy 패턴, Template Method 패턴).
> 
>- L: LSP (Liskov Substitution Principle) - 리스코프 치환 원칙
>    - 핵심: 서브타입(자식 클래스)은 언제나 자신의 슈퍼타입(부모 클래스)으로 교체될 수 있어야 한다.
>    - 목표: 상속 관계에서 부모 클래스의 객체를 자식 클래스의 객체로 대체해도 프로그램의 정확성이 유지되도록 보장한다. 즉, 자식 클래스는 부모 클래스의 **행위 규약(Contract)**을 위반해서는 안 된다.
>    - 예시: Square 클래스가 Rectangle을 상속받을 때, Square의 너비를 변경하면 높이도 변경되는 행위가 Rectangle의 기본 행위를 위반하지 않아야 한다.
> 
>- I: ISP (Interface Segregation Principle) - 인터페이스 분리 원칙
>    - 핵심: 클라이언트는 자신이 사용하지 않는 메소드에 의존해서는 안 된다. 즉, 인터페이스는 작고 구체적이어야 한다.
>    - 목표: 거대하고 범용적인 인터페이스(Fat Interface)를 여러 개의 작은 역할별 인터페이스로 분리하여, 구현 클래스가 불필요한 메소드를 강제로 구현하지 않도록 한다.
>    - 예시: 모든 기능을 담은 WorkerInterface 대신, Eatable, Workable 등 역할별 인터페이스를 분리한다.
>
>- D: DIP (Dependency Inversion Principle) - 의존성 역전 원칙
>    - 핵심:
>        - 고수준 모듈(High-level module)은 저수준 모듈(Low-level module)에 의존해서는 안 된다. 둘 다 추상화에 의존해야 한다.
>        - 추상화(인터페이스)는 세부 사항(구현 클래스)에 의존해서는 안 된다. 세부 사항이 추상화에 의존해야 한다.
>    - 목표: 시스템의 설계가 변경하기 쉬운 구체 클래스에 묶이는 것을 방지하고, 변경하기 어려운 추상화(인터페이스)에 의존하게 만든다.
>    - 구현 방법: 의존성 주입(DI)을 통해 고수준 모듈이 저수준 모듈의 구체 클래스를 직접 생성하거나 참조하는 것을 막고, 인터페이스를 통해 느슨하게 결합시킨다.
